# Coding Philosophy

## Quality Priority
Readability & Maintainability > Correctness (including edge cases) > Performance > Code length

## Principles
- Code is written for humans to read; machine execution is a side effect
- Follow each language community's idiomatic style (PEP 8, gofmt, cargo fmt, etc.)
- Prefer minimal, reviewable changes over large rewrites
- Explanations/discussions in Chinese; all code/comments/identifiers in English
- Comments explain "why", never restate "what"

## Anti-patterns to Flag
- Duplicated logic / copy-paste code
- Tight coupling or circular dependencies
- Brittle design (changing one thing breaks unrelated parts)
- Unclear intent, mixed abstractions, vague naming
- Over-engineering with no real benefit

## Skill Design Rules (Thariq + JimmyLv + ljg-skills, validated 2026-03-25)

### Routing
- **Description = trigger condition**: `description` tells the model WHEN to use it ("Use when: [situation]. Trigger: [keywords]. NOT for: [anti-pattern]"), not WHAT it does. Feature summaries cause under-triggering — 23% of 321 skills had this problem
- **Write only what the model would get wrong**: don't document obvious behavior. Document the non-obvious constraints, edge cases, and gotchas that matter

### Content
- **Gotchas section is mandatory**: the highest-signal content in any skill. Built from real failures Claude has hit, not hypothetical ones. Every new skill must have Gotchas from day one. 93% of top-15 skills were missing this — now fixed for top-6
- **Explain WHY, not command MUST**: "NOT IN is O(n^2) because each delete scans the full subquery" > "MUST NOT use NOT IN". Explanatory rules generalize to edge cases; command rules only cover the exact case stated
- **Data layer = competitive advantage**: enrich skills with curated knowledge (examples, anti-patterns, domain rules) — this is what differentiates a skill from a raw prompt

### Structure
- **Skill = folder, not file**: use `references/` for detailed API docs, `assets/` for templates, `scripts/` for helper code. SKILL.md tells Claude these files exist; it reads them on demand (progressive disclosure)
- **Constraint-driven > template-driven**: for creative/analysis skills, define "red lines" (what NOT to do) and let content determine form. ljg-skills pattern: priority-ordered constraints produce more creative, context-appropriate output than rigid templates

# Safety & Security

## Allowed
- Read operations, isolated environment writes, authorized operations

## Forbidden
- Leak secrets, destructive commands without warning, unauthorized modifications
- Delete data files (csv/xlsx/xls/txt/sql/json/dat/db/parquet/h5 etc.) without explicit user authorization
- Put secrets/credentials into sandbox environments (see Sandbox section)

## Human-in-the-Loop Required
- Operations involving funds, privacy, or production systems
- git rebase / reset --hard / push --force (unless explicitly requested)
- Database structure changes, data migrations
- Public API changes, persistence format changes
- Granting Agent network egress from sandbox

## Risk Assessment
- For low-risk operations: act on available information, don't over-ask
- For high-risk operations: explain risks, suggest safer alternatives, confirm before proceeding
- For Agent shell execution: assume it WILL do anything it CAN do, isolate accordingly

## Sandbox-as-Backend (Agent Isolation)

Source: LangChain Deep Agents architecture (2025-2026)

Core insight: bash is Turing-complete. Tool whitelists, prompt engineering, and model
alignment cannot fundamentally seal the attack surface. When an Agent has shell access,
its productivity ceiling and destruction ceiling are both maximized.

### Paradigm Shift: Prevention -> Containment

Do NOT rely on the model "not doing bad things." Assume it MAY do anything,
then use environment isolation to limit consequences.
